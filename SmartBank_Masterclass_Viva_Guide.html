Chapter 1: The Big Picture (Architecture)

Before diving into code, you must be able to explain how your project works from a bird's-eye view. SmartBank follows a classic Client-Server Architecture.

The Three Pillars of SmartBank
  • The Frontend (Client): Written in pure HTML, CSS, and JavaScript. This runs directly inside the user's browser (Chrome/Edge). Its only job is to display UI and send requests to the backend.
  • The Backend (Server): Written in Python using the Flask framework. This runs on the server (e.g., Render). It receives requests from the frontend, enforces security rules, and talks to the database.
  • The Database: SQLite (for local) or PostgreSQL (for production). It permanently stores users, accounts, and transactions on the server's hard drive.

How they talk (The API)

Your frontend and backend talk to each other using an API (Application Programming Interface) via the HTTP protocol.

  • Frontend asks: "Hey Backend, here is a username and password. Are they correct?" (This is an HTTP POST request to `/api/auth/login`).
  • Backend checks the Database.
  • Backend replies: "Yes, here is a success message and a session cookie." (This is an HTTP JSON Response).
Viva Tip: If the examiner asks "Why use Flask and not Django?", reply: "Flask is a micro-framework. It gave me the absolute control to build exactly what I wanted from scratch without unnecessary bloat, allowing me to fully understand the flow of data."

Chapter 2: The Backend & Database (Python + SQL)

Let's look at how the Python code actually works.

1. The Entry Point: main.py

This file is the beating heart of your server.

from flask import Flask, request, jsonify
from blueprints.auth_routes import auth_bp

# 1. We create the Flask application object
app = Flask(__name__)

# 2. We set a secret key for security (cookies/sessions)
app.secret_key = 'super_secret_key'

# 3. We register Blueprints (Modules)
app.register_blueprint(auth_bp, url_prefix='/api/auth')

# 4. We start the server
if __name__ == '__main__':
    app.run(debug=True, port=5000)

What is a Blueprint? Imagine writing 10,000 lines of code in `main.py`. It would be a nightmare. Flask Blueprints allow you to split your app into separate files (auth, users, staff, transactions) and snap them together like Lego blocks in `main.py`.

2. Handling a Request (auth_routes.py)

Here is what happens when a user tries to log in.

@auth_bp.route('/login', methods=['POST'])
def login():
    # 1. Get the JSON data sent by the Frontend
    data = request.json
    username = data.get('username')
    password = data.get('password')

    # 2. Connect to the Database
    db = get_db()
    cursor = db.cursor()

    # 3. Execute an SQL Query to find the user
    cursor.execute("SELECT * FROM users WHERE username = ?", (username,))
    user = cursor.fetchone()

    # 4. Check if user exists and password matches
    if user and check_password_hash(user['password'], password):
        session['user_id'] = user['id'] # Create a login session
        return jsonify({"success": True, "message": "Logged in!"})

    return jsonify({"error": "Invalid credentials"}), 401
Key Python Concepts Used Here:
  • Decorators (`@auth_bp.route`): This tells Flask, "Whenever a user visits `/login` via a POST request, please run the `login()` function directly below this."
  • Dictionaries: `data` is a Python Dictionary. We use `.get('username')` to safely extract values without crashing if the key doesn't exist.
  • Tuples in SQL: Notice the `(username,)`? We pass SQL variables as Tuples. This prevents SQL Injection attacks (where hackers try to delete your database via the login box).
  • Hashing: We never save plain text passwords in the database. We hash them. `check_password_hash` compares the hash to what the user typed.

Chapter 3: The Database (SQL & Transactions)

The database is structured using Relational Tables. The most important concept here is ACID Properties.

The Money Transfer Example

If User A sends ₹500 to User B, two things must happen in the database:

  1. Deduct ₹500 from Account A.
  2. Add ₹500 to Account B.

What if the server crashes exactly between Step 1 and Step 2? The money vanishes! To prevent this, we use Database Transactions (Commit and Rollback).

try:
    # Deduct from sender
    cursor.execute("UPDATE accounts SET balance = balance - 500 WHERE id = 1")
    # Add to receiver
    cursor.execute("UPDATE accounts SET balance = balance + 500 WHERE id = 2")

    # If both succeed, COMMIT (save) permanently
    db.commit()
except Exception as e:
    # If ANY error happens, ROLLBACK (cancel everything)
    db.rollback()

This guarantees Atomicity (the 'A' in ACID)—meaning the transaction is 'all or nothing'.

Primary vs Foreign Keys

In `schema.sql`, the `users` table has an `id` column. That is the Primary Key (unique identifier for the user). The `transactions` table has a `user_id` column. That is the Foreign Key (it points back to the user's Primary Key so we know who made the transaction).

Chapter 4: The Frontend (HTML, CSS, JS)

Now let's look at the browser side. How does the frontend actually talk to that Python code?

1. The JavaScript (Fetch API & Async/Await)

In `user-auth.js`, we have functions that trigger when the user clicks "Login".

async function handleLogin(event) {
    event.preventDefault(); // Stop the page from reloading

    // 1. Grab what the user typed in the HTML boxes
    const username = document.getElementById('loginUsername').value;
    const password = document.getElementById('loginPassword').value;

    try {
        // 2. Send the data to the Python Backend
        const response = await fetch('/api/auth/login', {
            method: 'POST',
            headers: { 'Content-Type': 'application/json' },
            body: JSON.stringify({ username: username, password: password })
        });

        // 3. Wait for Python to reply
        const data = await response.json();

        if (data.success) {
            window.location.href = 'userdash.html'; // Redirect!
        } else {
            alert(data.error);
        }
    } catch (error) {
        console.error("Network failed", error);
    }
}
Why Async/Await?

JavaScript runs on a single thread. If you send a request over the internet to a server, it takes a few milliseconds. If JS waits blindly, the entire browser freezes (you couldn't even click other buttons). async/await tells JS: "Send the request, go handle other things on the website, and come exactly back to this line of code when the server replies."

2. DOM Manipulation

The Document Object Model (DOM) is how JS interacts with HTML. When you write `document.getElementById('balanceDisplay').innerHTML = "₹500";`, you are directly modifying the HTML tree in real-time, causing the screen to instantly update without reloading the page. This is what makes SmartBank feel like an "app" rather than an old website.

Chapter 5: HTML & CSS Deep Dive

HTML: Semantic Tags

In `staffdash.html`, you didn't just use `<div>` for everything. You used `<header>`, `<nav>`, `<main>`, and `<aside>`.

  • Why? This is called Semantic HTML. It tells the browser exactly what the content is. Screen readers for blind users rely on this to navigate the page correctly. Search engines (Google) use this to understand the webpage structure.

CSS: The Box Model

Every single element on your website is a rectangular box. The Box Model consists of:

  • Content: The actual text or image.
  • Padding: The invisible space inside the box (between the text and the border).
  • Border: The line around the box.
  • Margin: The invisible space outside the box (pushing other boxes away).

CSS: Flexbox vs Grid

You used both heavily in `modern-dashboard.css`.

  • Flexbox (`display: flex`): Used for 1-Dimensional layouts. Like the Top Navigation bar. You want items lined up in a row, evenly spaced (`justify-content: space-between`), and vertically centered (`align-items: center`).
  • CSS Grid (`display: grid`): Used for 2-Dimensional layouts. Like the Dashboard Cards. You want a grid that is 2 columns wide on desktops, but automatically shrinks to 1 column on mobile phones (`grid-template-columns: 1fr 1fr`).
Viva Tip: When asked how your site is Mobile Responsive, say: "I used CSS Media Queries (`@media (max-width: 768px)`) to detect small screens and automatically transform the CSS Grid from multiple columns into a single vertical column for easy scrolling."
