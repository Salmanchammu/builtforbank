Chapter 4: Databases and the Flask Framework

4.1 What is a Database?

If variables in JavaScript and Python act as memory boxes, why do we need a database? Because variables exist in RAM (Random Access Memory). When the server restarts or loses power, RAM is wiped clean. A database is a specialized software that saves data persistently to a hard drive so it survives reboots. For a banking app, losing data is not an option.

4.2 Relational (SQL) vs. Non-Relational (NoSQL)

There are two major families of databases in modern web development.

  • NoSQL (e.g., MongoDB): Data is stored as flexible, JSON-like documents. You might have one document for a user that contains their name, age, and a nested list of their transactions. It's highly flexible and great for unstructured data or rapidly changing prototypes.
  • Relational SQL (e.g., SQLite, PostgreSQL): Data is stored in strict, rigid tables with rows and columns. SmartBank uses a Relational Database. Why? Because banking data is highly structured and interconnected. A User has an Account. An Account has Transactions. SQL is built specifically to handle these relationships safely and efficiently.

4.3 The Anatomy of SQL Tables (Primary and Foreign Keys)

In your `schema.sql` file, you define tables. Think of a table like an Excel spreadsheet. Every column has a specific data type (e.g., INTEGER, TEXT).

The Primary Key

Every single row in a table must have a unique identifier. This is the Primary Key. In your `users` table, the `id` column is the Primary Key. Even if two users are named "Salman", their IDs will be `1` and `2`. The database guarantees that no two rows will ever have the same Primary Key.

The Foreign Key

This is how relationships are formed. Suppose you have a `transactions` table. If you want to record that Salman deposited ₹500, you don't write "Salman" in the transactions table. You write `1` (Salman's Primary Key ID) in a column called `user_id`. In this context, `user_id` is a Foreign Key because it points to the Primary Key of a foreign table. This links the two tables together.

4.4 SQL Commands (CRUD)

You interact with the database using SQL (Structured Query Language). Most operations fall under the acronym CRUD:

  • Create: INSERT INTO users (name, role) VALUES ('Salman', 'Admin')
  • Read: SELECT * FROM users WHERE id = 1
  • Update: UPDATE accounts SET balance = 5000 WHERE id = 1
  • Delete: DELETE FROM transactions WHERE status = 'failed'

4.5 ACID Properties (Crucial for Viva!)

For a banking application, the database must be ACID compliant. This guarantees that your money is safe even during server crashes.

  • Atomicity: "All or Nothing." If you transfer ₹500 to a friend, the database must deduct ₹500 from you AND add ₹500 to your friend. If the server crashes in the exact millisecond between those two actions, Atomicity guarantees the database will automatically cancel the entire transaction. The money doesn't disappear.
  • Consistency: The database rules are strictly enforced. If a column is defined as an INTEGER, trying to save the word "Hello" will be rejected instantly.
  • Isolation: If two people try to transfer money at the exact same microsecond, Isolation ensures they are processed sequentially, so the math doesn't overlap and calculate the balance incorrectly.
  • Durability: Once the database says "Transaction Successful", the data is permanently written to the hard drive. If someone kicks the server's power plug out a millisecond later, the data is still there when the server reboots.

4.6 The Flask Web Framework

Now that we have HTML/JS on the frontend and a Database on the backend, how do they talk? Enter Flask.

Flask is a web framework written in Python. A framework is simply a large collection of pre-written code that handles the boring, repetitive parts of web development (like processing HTTP headers and managing URLs), so you can focus on writing your actual app logic.

WSGI (Web Server Gateway Interface)

Python does not inherently know how to speak the language of the internet (HTTP). Web servers like Apache or Nginx speak HTTP, but they don't understand Python code. WSGI is the universal translator sitting in the middle. Flask uses Werkzeug, a WSGI utility library, to translate web requests into Python objects that your code can understand.

Micro-Framework vs. Batteries Included

You will likely be asked, "Why Flask and not Django?"

Django is a "batteries included" framework. It comes with its own massive ORM (database system), an admin panel, and an authentication system pre-installed. You just configure it.
Flask is a "micro-framework". It gives you the absolute bare minimum required to get a server running. Everything else—database connections, login logic—you have to write yourself. You chose Flask because it gave you absolute architectural freedom and allowed you to learn exactly how full-stack systems work from the ground up, rather than relying on automated magic.

Routing

Routing is how Flask knows which Python function to run when a user visits a specific URL.

@app.route('/dashboard', methods=['GET'])
def show_dashboard():
    return render_template('dashboard.html')

When the client's browser requests the URL /dashboard, Flask matches that URL to this specific decorator and executes the show_dashboard() function. The function then reads the HTML file from the hard drive and sends it back to the browser over the internet.
