5.1 The Entry Point: backend/main.py

This is the file you run (python main.py) to start your entire server. It orchestrates everything.

from flask import Flask, request, jsonify
from flask_cors import CORS
from blueprints.auth_routes import auth_bp

app = Flask(__name__)
CORS(app)
app.secret_key = 'super_secret_key'

app.register_blueprint(auth_bp, url_prefix='/api/auth')

if __name__ == '__main__':
    app.run(debug=True, port=5000)

Line-by-Line Explanation:

  • from flask import Flask: We import the main Flask class from the library we installed via pip.
  • from flask_cors import CORS: CORS (Cross-Origin Resource Sharing) is a massive security feature. By default, browsers block a frontend running on localhost:3000 from talking to a backend on localhost:5000 to prevent hackers from stealing data. We import CORS to explicitly tell the browser, "It's okay, I own both of these, let them talk."
  • app = Flask(__name__): We create an instance (an Object) of the Flask class. This app variable is your web server.
  • CORS(app): We apply the CORS rules to our server.
  • app.secret_key = '...': Flask uses this cryptographic key to sign session cookies. If a user tries to forge a cookie to pretend they are the admin, Flask checks the signature against this key and rejects it.
  • app.register_blueprint(auth_bp, url_prefix='/api/auth'): Instead of writing 10,000 lines of code in main.py, we split our app into modules (Blueprints). Here, we import the Authentication module and tell Flask: "Any time a user requests a URL starting with /api/auth, send them to this blueprint."
  • if __name__ == '__main__':: This ensures the server only starts if we run main.py directly, not if we just import it into another script.
  • app.run(debug=True, port=5000): This starts the server on port 5000. debug=True means if our code crashes, it prints a helpful error message to the browser instead of a generic 500 Error. It also automatically restarts the server if we save changes to the code.

5.2 The Frontend Communication: user-auth.js

Now, let's look at what happens in the browser when the user clicks the "Login" button. We use Vanilla JavaScript.

async function handleLogin(event) {
    event.preventDefault();

    const username = document.getElementById('loginUsername').value;
    const password = document.getElementById('loginPassword').value;

    try {
        const response = await fetch('/api/auth/login', {
            method: 'POST',
            headers: { 'Content-Type': 'application/json' },
            body: JSON.stringify({ username: username, password: password })
        });

        const data = await response.json();

        if (data.success) {
            window.location.href = 'userdash.html';
        } else {
            alert(data.error);
        }
    } catch (error) {
        console.error("Network failed", error);
    }
}

Line-by-Line Explanation:

  • async function: We declare the function as asynchronous because it will be making a network request over the internet, and we don't want to freeze the browser while waiting.
  • event.preventDefault(): By default, when you click a submit button inside an HTML form, the browser tries to refresh the entire page. We cancel this default behavior so we can handle the submission smoothly with JavaScript instead.
  • document.getElementById(...).value: We use the DOM to grab the HTML input boxes by their unique IDs and extract the text the user typed inside them.
  • try { ... } catch (error) { ... }: We wrap our network code in a try/except block. If the user loses their WiFi connection exactly when they click login, the fetch will fail. Instead of crashing the script, the catch block gracefully logs the network error.
  • await fetch('/api/auth/login'): The Fetch API is built into modern browsers. It is how JS sends HTTP requests. We await it, meaning JS pauses execution on this specific line until the Python server replies, but allows the rest of the browser to keep running.
  • method: 'POST': We are sending data (username/password) to the server to create a session, so we use the HTTP POST method.
  • headers: { 'Content-Type': 'application/json' }: We explicitly tell the Python server, "Hey, the data I'm sending you is formatted as JSON."
  • body: JSON.stringify(...): We take our JavaScript Object containing the username and password and convert it into a pure text JSON string so it can travel across the internet.
  • const data = await response.json(): The Python server replies with a JSON string. We use .json() to parse that string back into a usable JavaScript Object.
  • if (data.success): If Python replied with {"success": true}, we use window.location.href to redirect the browser to the dashboard page.

5.3 The Database Logic: core/db.py

Finally, how does Python actually execute queries?

import sqlite3
from flask import g

def get_db():
    if 'db' not in g:
        g.db = sqlite3.connect('database/smartbank.db')
        g.db.row_factory = sqlite3.Row
    return g.db

def close_db(e=None):
    db = g.pop('db', None)
    if db is not None:
        db.close()

Line-by-Line Explanation:

  • import sqlite3: We import the built-in library that allows Python to talk to SQLite databases.
  • from flask import g: In Flask, g stands for Global Context. It is a special object that exists only for the duration of a single HTTP request. It's used to store temporary data that multiple functions might need while processing that one specific request.
  • if 'db' not in g:: We check if a database connection has already been opened for this user's request. Why? Because opening a database connection is slow and resource-heavy. We don't want to open 5 connections if we run 5 queries in a single request. This is called the Singleton Pattern.
  • g.db = sqlite3.connect(...): We open the connection to the database file on the hard drive and store the connection object in the g context.
  • g.db.row_factory = sqlite3.Row: By default, SQLite returns rows as raw Tuples (1, "Salman", 500). By setting this row factory, SQLite returns rows that act like Dictionaries {"id": 1, "name": "Salman", "balance": 500}. This makes the code vastly more readable because we can access columns by name instead of index numbers.
  • def close_db(e=None):: This function is triggered by Flask the exact millisecond an HTTP request finishes and the response is sent to the user.
  • db = g.pop('db', None): We safely extract the database connection from the global context.
  • db.close(): We explicitly close the connection. This is absolutely critical. If you don't close connections, the database eventually locks up and the entire server crashes.
